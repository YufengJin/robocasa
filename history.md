### Known limitations

- **Sparse reward only** — `Kitchen.reward()` returns 0/1 success. Anyone trying to fit a value function on raw reward will need to add their own shaping; the benchmark is designed for BC / offline-RL on demonstrations, not online RL.
- **Random policy returns 0% success** — expected. Smoke quality is measured by the WebSocket handshake + obs/action shape contract, not by policy task success.
- **Demonstrations are not in the image** — to BC-train, the user must run `python robocasa/scripts/download_datasets.py --tasks <task>` separately (see `docs/use_cases/downloading_datasets.md`); kitchen *assets* (~5GB) are pulled by entrypoint on first boot.
- **`--arm_controller joint_pos` / `joint_vel` are flagged as BETA** by the upstream scripts; only `cartesian_pose` (7-D OSC pose) is the official RoboCasa benchmarking contract. Use `joint_vel` (8-D) only for OpenPI / DROID compatibility runs.
- **Mobile base dimensions** — env action dim is 11 (6 OSC pose + 1 gripper + 1 torso + 3 mobile base) for the default `PandaMobile` robot; `pad_action_for_env` zero-pads policy output (7-D for cartesian_pose) to env-facing 11-D. Switching to a fixed-base robot will change `env.action_dim`; the policy-facing `action_dim` is unchanged.
