### Known limitations

- **`network_mode: host` port collisions**: the compose profile shares the host's port namespace. During this run port 8000 was already bound on the host by a stale `test_random_policy_server.py` (left over from an earlier session, PID 1673800 under a different user, not reachable from the container namespace anyway under host networking). The smoke test switched to port 8001. On a fresh host this collision does not occur and port 8000 (the script default) works as documented in `install.md`.
- **Sparse success reward only**: `Kitchen.reward()` returns 0 or 1. Reward-based debugging of a random policy is not possible — the signal is always 0 until the task actually succeeds. This is inherent to robocasa, not a build issue.
- **Kitchen assets are not shipped in the image**: the ~5 GB textures are pulled on first container boot into the host-mounted `robocasa/models/assets/textures/`. Deleting the mount discards the assets and triggers a re-download on next `up`.
- **mimicgen + robosuite_models are absent**: Dockerfile does not install `mimicgen` (prints a non-fatal warning on every import) or `robosuite_models` (extra robot zoo). Neither is required for the default `--task_name PnPCounterToCab` + `--robots PandaMobile` smoke path, but tasks or robots that require them will fail with a clear `ImportError`.
- **`env.action_dim=11` vs. `ACTION_DIM=7`**: the raw robosuite env exposes 11-D actions for `PandaMobile` (arm 6 + gripper 1 + torso 1 + base 3). The policy server advertises 7-D (arm 6 + gripper 1); `scripts/run_demo.py` pads the remaining four dims with zeros via `pad_action_for_env`. This is intentional and matches the OpenVLA 7-D policy output expectation; do **not** switch the server to 11-D without changing `--arm_controller` accordingly.
- **No training script is generated**: IL BC training is benchmark-specific and upstream expects you to use [robomimic](https://github.com/ARISE-Initiative/robomimic) against collected demos; see `benchmark.md` for the recommended path.
