## About

RoboCasa is a large-scale kitchen-environment benchmark built on top of [robosuite](https://github.com/ARISE-Initiative/robosuite) and MuJoCo. It provides 120 procedurally-generated kitchen layouts and 25+ atomic manipulation tasks plus 5 long-horizon multi-stage tasks intended for imitation learning and offline RL research. The paper is Nasiriany, Maddukuri, Zhang et al. (RSS 2024); the project page is <https://robocasa.ai>.

### Task groups

Tasks cluster along the manipulation primitive required. **Pick-and-place** (8) exercise the default 6-DOF OSC pose controller with a gripper open/close toggle. **Articulated** (door / drawer / faucet / stove / microwave, 11) require reaching and actuating a revolute or prismatic joint on fixed kitchen hardware. **Coffee** (3) and **mobile navigation** (1) are one-off task families. The 5 **multi-stage** tasks chain two or more of the above primitives in sequence and use longer horizons; they are the intended benchmark for long-horizon IL methods.

## How to use

The skill produced three receipt files at the repo root â€” `install.md` (how to build), `history.md` (what the skill did), and this file (what the benchmark *is*). The flow below assumes you have already followed `install.md` and have a running container.

### 1. Enter the container

```bash
docker compose -f docker/docker-compose.headless.yaml up -d
docker exec -it robocasa_container bash
```

### 2. Pick a task

```bash
# List available tasks (shell one-liner into the task registry)
python -c "from robocasa.utils.dataset_registry import SINGLE_STAGE_TASK_DATASETS, MULTI_STAGE_TASK_DATASETS; print('\n'.join(list(SINGLE_STAGE_TASK_DATASETS) + list(MULTI_STAGE_TASK_DATASETS)))"
```

Any identifier from the task table above is a valid value for `--task_name`.

### 3. Stand up a policy server

For a smoke test, use the companion random-policy server (ships with the image):

```bash
python tests/test_random_policy_server.py --port 8765
```

For real evaluation, replace it with any server that speaks the `policy_websocket` protocol and advertises `action_dim=7` (cartesian_pose) or `action_dim=8` (joint_pos / joint_vel) in its metadata handshake â€” the client's guard in `scripts/run_demo.py` / `scripts/run_eval.py` enforces the match against `--arm_controller`.

### 4. Run a single-episode demo

```bash
python scripts/run_demo.py --policy_server_addr localhost:8765 \
    --task_name PnPCounterToCab --num_resets 1 --arm_controller cartesian_pose
```

This is the "show me one rollout" entry point â€” per-step inference log, one MP4 per episode saved under `--demo_log_dir` (default `./demo_log`).

### 5. Run multi-episode eval

```bash
python scripts/run_eval.py --policy_server_addr localhost:8765 \
    --task_name PnPCounterToCab --num_trials 10 --arm_controller cartesian_pose
```

Prints rolling success rate and final aggregate across `--num_trials` rollouts. Logs are persisted to `<log_dir>/<task>--<YYYYMMDD_HHMMSS>/eval.log`. Add `--no_save_video` for a fast eval loop; omit it to keep one MP4 per rollout.

Success semantics are task-specific but uniform: the env's `_check_success()` predicate returns `True` as soon as the scene satisfies the task goal (e.g., object-in-container contact test for PnP tasks), at which point `reward` transitions from 0 to 1.

### 6. Train / collect demonstrations

For behavior cloning on this benchmark you will typically:

1. Download the official demonstration datasets from the links in `robocasa/utils/dataset_registry.py` (Human-50 for human teleoperated demos, or the Machine-Generated MimicGen dataset for scale).
2. Train your policy off-line against those demos â€” the upstream recipe uses [robomimic](https://github.com/ARISE-Initiative/robomimic) (install separately; not in this image).
3. Wrap the trained policy in a `policy_websocket`-speaking server (mirrors `tests/test_random_policy_server.py` â€” subclass `PolicyWebsocketServer`, advertise `action_dim` matching the controller, implement `inference(obs_dict)`).
4. Point `scripts/run_eval.py --policy_server_addr host:port` at that server to measure success rate across tasks, seeds, and object-instance splits (`--obj_instance_split` = `A` train / `B` held-out).

The skill does **not** generate a training script for IL benchmarks â€” BC training code is benchmark-specific and upstream usually ships its own. See <https://github.com/ARISE-Initiative/robomimic> for the officially recommended training entry point.

## Troubleshooting

- `ConnectionRefused` on `run_demo.py` / `run_eval.py` â†’ no policy server on the configured port. Start `tests/test_random_policy_server.py` first (IL) or check the server process logs.
- `ValueError: arm_controller=... expects policy action_dim=X, but server returns Y` â†’ the policy server advertises a different action dim than `--arm_controller` expects. Use `cartesian_pose` with a 7-D server (OpenVLA-style) or `joint_vel` / `joint_pos` with an 8-D server (OpenPI-style).
- Task name not recognised â†’ check the task table above; identifiers are case-sensitive and follow the `CamelCase` convention.
- GUI apps fail with `cannot open display` â†’ you are on the headless compose profile. Switch to `docker-compose.x11.yaml` after `docker compose down`.
- `[Errno 98] Address already in use` on the server â†’ the compose profile uses `network_mode: host`, so container ports share the host namespace. Either free port 8765 on the host or pass `--port 8001` to the server and `--policy_server_addr localhost:8001` to the clients.
- Training dataset hashes or paths drift â†’ `robocasa/utils/dataset_registry.py` is the source of truth for `hf_hub_url` and `dataset_type`; re-pull from HuggingFace when upstream revs.
